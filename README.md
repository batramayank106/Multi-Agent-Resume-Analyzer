## Quick Start

```bash
# 1. Clone
git clone https://github.com/batramayank106/Multi-Agent-Resume-Analyzer.git
cd Multi-Agent-Resume-Analyzer

# 2. Configure secrets (edit with your own API keys)
cp .env.example .env

# 3. Run with Docker (recommended)
docker compose up --build

# Or run manually:
# pip install -r requirements.txt
# python run.py all
```

Then open **http://localhost:8501** in your browser.

> **Zero API keys needed** if you have Ollama running locally — set `PREFER_OLLAMA=true` in `.env`. See [Ollama setup](#option-c-ollama-local-llm--no-api-keys-required) below.

## The 12-Agent Pipeline

CV Chacha orchestrates **12 specialized agents** in a sequential LangGraph pipeline. Each agent receives the full shared state and contributes structured outputs:

```
Resume + JD
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  1. ATS Scoring Agent          (rule-based · zero API cost)   │
│  2. Recruiter Review Agent     (Groq LLM · readability)       │
│  3. Hiring Manager Agent       (Groq LLM · technical depth)   │
│  4. Recruiter Simulator        (Groq LLM · 3 personas)        │
│  5. Skill Gap Agent            (Groq LLM · missing skills)    │
│  6. Truthfulness Agent         (Groq LLM · claim validation)  │
│  7. Company Research Agent     (Groq LLM · culture/fit)       │
│  8. Salary Estimation Agent    (Groq LLM · role/company/loc)  │
│  9. Resume Rewrite Agent       (Groq LLM · STAR opt.)         │
│ 10. Interview Prep Agent       (Groq LLM · question gen.)     │
│ 11. Famous Questions Agent     (Groq LLM · company-known Qs)  │
│ 12. Selection Probability      (Groq LLM · hiring odds)       │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
Structured Analysis Report (16 visual pages)

All LLM calls go through Groq first (fast, free, 30 RPM), then fall back to Hugging Face API → Ollama → graceful degradation.
```

Each agent is implemented as a pure function node in a `StateGraph` with typed `AgentState`. If any agent fails, the pipeline continues with a graceful error — no single point of failure.

## Deep Dive: 12 Agents Explained

### 1. ATS Scoring Engine (Rule-Based)
The only non-LLM agent — designed this way intentionally for consistency, speed, and zero API cost.

**Weights:** Keyword Match 30% · Skill Match 20% · Experience 15% · Project 15% · Achievements 10% · Education 5% · Formatting 5%

**Output:** 0–100 score with 5-tier label (Poor/Average/Good/Strong/Exceptional), per-category breakdown, and evidence-backed deductions for every point lost.

**Engineering challenge solved:** How do you make ATS scoring explainable? Every deduction stores the exact keyword that was missing, the JD context it appeared in, and the reason it was weighted. No unexplained black-box scores.

### 2. Recruiter Review (LLM)
Evaluates readability, action-verb density, impact orientation, leadership signals, and communication clarity. Powered by Groq (llama-3.3-70b-versatile) as primary, falls back to HF API / Ollama.

### 3. Hiring Manager Review (LLM)
Assesses technical depth, project complexity, domain-specific expertise, and growth trajectory. The prompt is engineered to mimic how actual hiring managers think — looking for specific technical signals.

### 4. Recruiter Simulator (3 Personas)
Runs **three independent LLM calls** in a single graph node:
- **ATS Persona**: Keyword compliance, formatting strictness
- **HR Persona**: Communication, leadership, presentation
- **Engineering Manager Persona**: Technical depth, project quality

Each returns PASS/CONDITIONAL/REJECT with confidence scores. The simulator aggregates them into an overall decision.

**Engineering challenge solved:** How do you make three LLM personas diverge meaningfully? Each persona uses a different system prompt with role-specific rubrics and scoring criteria.

### 5. Skill Gap Analysis (LLM)
Compares resume skills against JD requirements. Classifies into: required skills present, required skills missing, preferred skills present, preferred skills missing. Outputs gap severity (low/medium/high/critical).
- **Suggested Additions** — recommends 2-3 complementary skills not in the JD that would strengthen the candidate's profile, each with a relevance reason.

### 6. Truthfulness Validation (LLM)
Critically examines claims for exaggeration. Classifies each as Safe / Needs Verification / Potentially Misleading. Never invents skills or projects. Outputs a truthfulness score + flagged items with reasoning.

**Engineering challenge solved:** How do you prevent an LLM from hallucinating during truthfulness checks? The prompt explicitly forbids the model from claiming any skill or project exists. It can only flag what's in the text.

### 7. Company Research & Culture (LLM)
Extracts company name from the resume/JD, then analyzes: culture fit indicators, growth opportunities, red flags (turnover language, unrealistic requirements). Returns overall suitability rating.
**Enhanced with:** `culture_score` (0-100), `review_summary` with pros/cons, `work_life_balance` and `career_growth` ratings — giving a Glassdoor-like employee review perspective directly from LLM knowledge.

### 8. Salary Estimation (LLM)
Estimates expected salary range for the role × company × location. Pulls from LLM training data (well-known companies have good coverage). Outputs range, currency, confidence level, and influencing factors.
**Frontend display:** Shown as a prominent card on the Company Insights page, between Overview and the 4-column layout.

### 9. Resume Rewrite (LLM)
Generates optimized bullet points using STAR methodology. Never changes template, layout, fonts, or structure — only content. Outputs both the rewritten resume and a summary of suggestions.

**Engineering challenge solved:** How do you generate better content without hallucinating? The rewrite uses all prior agent outputs (ATS gaps, recruiter feedback, skill gaps) as context, ensuring suggestions are grounded in actual deficiencies.

### 10. Interview Prep (LLM)
Generates 10-14 questions across 6 categories: Technical, Behavioral, Project & Portfolio, System Design, Company-Specific, Resume-Based Recruiter. Each question has difficulty (Easy/Medium/Hard) and a rationale explaining why it's asked.

**Engineering challenge solved:** How do you make questions specific to a person? The prompt receives the candidate's actual resume text, GitHub/portfolio links, company name (extracted from JD), and detected role. It generates questions referencing specific projects and experiences.

### 11. Famous Questions Agent (LLM)
Retrieves well-known interview questions for the target company (Amazon LPs, Google system design, Meta behavioral, etc.). Uses LLM knowledge of famous tech interview patterns. Questions are grouped by source (e.g. "Google SD", "Amazon LPs") and include difficulty and round context.
**Frontend display:** "Famous Questions" tab on the Interview Prep page alongside the generated questions.

### 12. Selection Probability (LLM)
Estimates hiring likelihood across stages: ATS Pass → Recruiter Shortlist → Technical Interview → Onsite → Offer. Each stage gets a percentage range + rationale based on all prior agent scores.

## Tech Stack & Engineering Decisions

### Why These Technologies?

| Technology | Why It Was Chosen | What It Demonstrates |
|-----------|-------------------|---------------------|
| **LangGraph** | State machine-based agent orchestration | Graph-based AI workflows vs linear chains |
| **FastAPI** | Async-first, auto-docs, Pydantic validation | Production API design patterns |
| **ChromaDB** | Persistent vector store, no external infra | RAG with local embeddings |
| **Cross-encoder reranker** | 2nd-pass relevance scoring | Advanced RAG beyond simple similarity |
| **BAAI/bge-small-en-v1.5** | 384-dim CPU-friendly embeddings | Trade-off: speed vs accuracy |
| **Groq Cloud** | Fastest free inference — 30 RPM, 14K req/day, no credit card | Primary LLM provider. Llama 3.3 70B via custom LPU hardware. |
| **Hugging Face Inference API** | No GPU needed, 5 model fallback chain (Qwen2.5 7B, DeepSeek-R1 7B, Llama 3.1 8B, Qwen2.5 72B, Qwen2.5 Coder 7B) | Cost-efficient LLM deployment. Falls back when Groq is unavailable. |
| **Streamlit** | Fast UI prototyping, Python-native | Rapid frontend development |
| **st.navigation** | Native routing, no third-party deps | Cleaner than streamlit-option-menu |
| **Plotly** | Interactive charting library | 13 chart types in centralized factory module |
| **python-docx** | Lightweight DOCX generation | Server-side report export with python-docx |
| **Architecture Diagrams** | Rendered PNG diagrams | 5 system architecture PNGs |

### Key Architecture Decisions

1. **ATS rule-based (not LLM):** Consistency, zero API cost, fully explainable. Every deduction is a data structure, not a prompt output.
2. **Sequential (not parallel) pipeline:** Simpler error handling, deterministic ordering, agents can read prior results.
3. **Agent-level error isolation:** Each agent is wrapped in try/except. One agent failure doesn't crash the pipeline.
4. **No local 7B+ models:** All inference via Groq Cloud. Runs on 2GB RAM CPU machines. Falls back to HF API then Ollama.
5. **warm-canvas editorial theme over dark mode:** Stands out from the typical dark-mode AI dashboard. Anthropic-inspired cream/coral palette.
6. **st.navigation over streamlit-option-menu:** Native routing avoids component caching bugs. More maintainable.
7. **Session state for frontend data:** Simple state management without a database dependency for quick prototyping.

## Resume Vault & Analysis History

The **Resume Vault** transforms CV Chacha from a one-time ATS checker into a persistent **Resume Intelligence Platform**.

### How It Works

Every **Analyze Resume** click creates an **Analysis Session** — a recoverable workspace containing:

| Component | Description |
|-----------|-------------|
| **Session ID** | UUID identifying the complete analysis |
| **Resume** | The uploaded resume text |
| **JD** | The job description used for the analysis |
| **Agent Results** | All 10 agent outputs (ATS, Recruiter, HM, Simulator, Skill Gap, Truthfulness, Company, Rewrite, Interview, Selection) |
| **All Scores** | ATS, Recruiter, HM, Truthfulness, Selection Probability |

### Key Features

- **Resume Vault Page** — Session list with score overview, date, company, role; delete with confirmation
- **Session Viewer** — 7-tab detail viewer: ATS, Recruiter, HM, Simulator, Skill Gap, Company, Rewrite/Interview/Selection
- **Resume Timeline** — Visualize progress: Upload → Analysis → Optimization → Reanalysis → Final
- **Version Comparison** — Side-by-side diff with keyword, score, and bullet-point highlighting
- **Resume Library** — All uploaded resumes with encrypted storage, download, and delete
- **Data Retention** — Sessions persisted in DB, deletable by user
- **Analytics** — ATS Score Trend, Recruiter Score Trend, Selection Probability Trends (Plotly)

### Database Design

```
User
  └── Resumes (many)
         └── Analysis Sessions (many)
                ├── Analysis Results (one-to-one)
                ├── Resume Versions (many)
                └── Interview Questions (many)
```

## Security Features

CV Chacha implements production-inspired security suitable for a portfolio-grade AI platform:

### Authentication
- **bcrypt password hashing** — never store plaintext passwords
- **JWT Access Tokens** (30 min lifetime) — secure API authentication
- **Refresh Tokens** (7 day lifetime) — seamless session persistence
- **Token auto-refresh** — transparent to the user

### Authorization (RBAC)
| Role | Privileges | How to Become |
|------|-----------|---------------|
| **User** | Own resume/analysis CRUD | Register via the Login page and log in. Every new registration is a regular user by default. |
| **Admin** | User management, audit logs | Promoted by a Super Admin from the Admin Dashboard → User Management page. |
| **Super Admin** | Full system access, role management | Set `SUPER_ADMIN_EMAIL` + `SUPER_ADMIN_PASSWORD` in `.env` before first run. The app auto-seeds the super admin on startup. Log in with those credentials. |

### Data Protection
- **Fernet encryption** at rest for resume content and analysis results
- **UUID-based file storage** prevents filename enumeration
- **Backend-controlled access** — no direct file URLs

### API Protection
- **Rate limiting** — custom middleware: 60 requests/min per IP across all `/api/` routes
- **Prompt injection detection** — 6 regex patterns (system override, role switch, prompt leak, delimiter break, token theft, jailbreak) with score-based blocking
- **Login lockout** — 5 failed attempts → 15 min lock
- **Password policy** — 8+ chars, 1 uppercase, 1 lowercase, 1 number
- **Audit logging** — all auth, upload, analysis, and admin events logged (sensitive data excluded)

### LLM Safety
- **Output validation layer** — every LLM response validated against Pydantic schemas before display
- **Score range validation** — ATS 0-100, probability 0-100, confidence 0-1
- **Anti-hallucination** — no recommendation without evidence; never invent skills/projects/experience
- **Confidence scoring** — every agent output includes confidence_score, evidence_count, source_references
- **Low-confidence flagging** — outputs below 0.5 confidence highlighted for user review
- **Resume rewrite protection** — AI must not fabricate skills, projects, employment history, or achievements
- **Evidence-based analysis** — "Missing Skill: FastAPI → Evidence: Found in JD, Not Found in Resume"

### Secure File Upload
- **Allowed:** PDF, TEX
- **Validated:** extension, MIME type, file size (max 20MB)
- **Rejected:** unsupported formats, oversized files, malicious content

## Getting Started

### Prerequisites

- **Python 3.10+** (3.14 recommended)
- **pip** (bundled with Python)
- **Docker** (optional — for containerized setup)
- **Ollama** (optional — for local LLM inference without API keys)
- **API keys** — Get a Groq API key at [console.groq.com/keys](https://console.groq.com/keys) (recommended) or a HuggingFace token at [hf.co/settings/tokens](https://huggingface.co/settings/tokens) (fallback). Skip if using Ollama.

---

### Configuration

All settings are managed via `.env` file at the project root. Copy `.env.example` to `.env` and fill in your values.

#### Required: API Keys (if not using Ollama)

| Variable | Required | Purpose | How to Get |
|----------|----------|---------|------------|

### Option C: Ollama (Local LLM — No API Keys Required)

If you want to run without any cloud API keys, use Ollama for fully local inference:

**1. Install Ollama**

| Platform | Command |
|----------|---------|
| Windows | `winget install Ollama.Ollama` or download from [ollama.com](https://ollama.com) |
| macOS | `brew install ollama` |
| Linux | `curl -fsSL https://ollama.ai/install.sh \| sh` |

**2. Pull a model based on your hardware**

| Hardware | Recommended Model | Size | Command |
|----------|------------------|------|---------|
| 2GB RAM, CPU only | `qwen2.5:0.5b` | ~400MB | `ollama pull qwen2.5:0.5b` |
| 4GB RAM / 2GB VRAM | `qwen2.5:1.5b` | ~900MB | `ollama pull qwen2.5:1.5b` |
| 8GB+ RAM / 4GB+ VRAM | `qwen2.5:3b` | ~1.9GB | `ollama pull qwen2.5:3b` |
| 16GB+ RAM / 8GB+ VRAM | `deepseek-r1:7b` | ~4.7GB | `ollama pull deepseek-r1:7b` |

**3. Start Ollama**

```bash
ollama serve
```

**4. Configure CV Chacha**

In your `.env` file, set:

```env
PREFER_OLLAMA=true
OLLAMA_MODEL=qwen2.5:3b
OLLAMA_GPU_LAYERS=20   # Set to 0 for CPU-only, 999 for full GPU offload
```

> 💡 **GPU acceleration**: For NVIDIA GPUs, set `OLLAMA_LLM_LIBRARY=cuda_v12` before starting Ollama. Partial offloading (`OLLAMA_GPU_LAYERS=20`) works well on 2GB VRAM GPUs like MX150/GTX 1050.

**5. Run the app**

```bash
python run.py all
```

> No GPU required for basic use — all models run on CPU. First inference is slow (~10-30s) as the model loads, subsequent calls are faster (~2-5s).

## Personal Note

### What I Learned

Before this project, I had pieces of knowledge — some Python, a basic understanding of LLMs, vague familiarity with Streamlit — but I had never put them together into something real. CV Chacha forced me to connect those dots.

On the **backend**, I learned how to structure a LangGraph pipeline with multiple agents, how to handle failures so one broken agent doesn't crash everything, and how to build an LLM manager that tries providers in order — Groq first, then Hugging Face API, then Ollama as last resort. I set up ChromaDB for vector storage, learned what chunk size and retrieval depth actually do to search quality, and wired it all together with FastAPI.

On the **frontend**, I learned Streamlit beyond the basics — managing session state across pages, building custom themes, handling file uploads in different formats, and creating charts that actually communicate something useful. A lot of trial and error went into getting the UI to feel cohesive.

On the **infrastructure** side, getting LLMs to actually run was a series of problems I had to work through one by one. First, the Hugging Face API credits ran out mid-testing — so I added Ollama for local fallback. Then Ollama wouldn't use my GPU — spent days searching why the CUDA v13 backend wasn't working, eventually found my MX150 needs v12 instead. Then the model wouldn't fit in 2GB VRAM — had to figure out partial offloading (20 layers out of 36, any more crashes). Every fix revealed a new problem.

Then I added **Groq** as the primary provider — it's cloud-based, fast, and has a generous free tier. But that meant reworking the LLM manager to try three providers in sequence: Groq first, HF if Groq fails, Ollama as last resort. Getting the fallback chain right took more trial and error than writing the actual code — testing rate limits, timeout handling, making sure failures in one provider don't cascade to the agents. The agents themselves have their own fallbacks too, so if every LLM fails, they return rule-based estimates instead of crashing. It's not elegant, but it doesn't break.

I can see this project being useful in a few contexts:

The project gave me a real sense of what those roles involve, and I have a clearer idea of where I want to go next.
